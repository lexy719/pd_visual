# Registers — what kind of website this is

A register is the genre a page belongs to: what furniture it needs, how dense it runs, and what
separates a competent example from an exceptional one. Register is decided before layout. Getting it
wrong makes every later decision wrong — a product page paced like an essay fails no matter how
beautiful its type is.

## Register: saas-product — the page is a decision, not a story

tags: layout, narrative, register, saas, product, conversion, nav, density, structure

A SaaS product page exists to move a skeptical evaluator to one action, so it runs DENSE and
scannable, never editorial-sparse. Required furniture: a sticky nav carrying the logo, 3–5 links and
ONE call to action; a hero that SHOWS the product (real interface, not an abstract photo) beside a
specific claim; capability blocks tied to outcomes; social proof placed adjacent to the ask, not in a
distant testimonial ghetto; transparent pricing; a footer that acts as a site map. Exceptional
versions put the actual product in the first viewport, use one repeated CTA rather than five
competing ones, and write specifics ("deploys in 40 seconds") where basic ones write adjectives
("blazing fast"). The register fails when the hero describes instead of shows, when features are a
grid of vague nouns with generic icons, or when proof is unattributed.

## Register: editorial-story — the page is a sequence, not a menu

tags: layout, narrative, register, editorial, story, narrative, brand, immersive, pacing

An editorial brand story earns attention scene by scene, so it runs SPARSE and sequential, with
little or no chrome — often just a minimal masthead, never a feature-stuffed nav. Required
furniture: an opening that commits to one image or one sentence; ordered narrative beats where each
section advances a single idea; deliberate pacing contrast between dense passages and quiet ones; a
close that resolves rather than converts. Exceptional versions vary section shape as the story
turns, use full-bleed media as punctuation rather than decoration, and let one moment be visually
unforgettable while everything else stays restrained. It fails when it becomes a scrapbook of
unrelated pretty sections, when every section is the same shape, or when it borrows SaaS furniture
(feature grids, pricing tables) that the narrative never needed.

## Register: local-service-business — trust before persuasion

tags: layout, narrative, register, local, service, clinic, trust, practical, contact

A local service business page (clinic, practice, workshop, studio) is read by someone deciding
whether to trust a real place with something they care about, so practical certainty outranks
atmosphere. Required furniture: what is offered, in plain words; who provides it, with real faces
and names; what it costs or how cost is handled; where it is and how to reach it, findable without
scrolling to the end; proof from actual clients. Exceptional versions answer the specific worry the
visitor already has before selling anything, name prices or price behaviour instead of hiding them,
and make the humans genuinely visible. It fails when it hides contact details in a footer, when it
photographs the building instead of the people, or when it adopts a corporate voice a two-person
practice would never use.

## Register: portfolio-showcase — the work argues, the page gets out of the way

tags: layout, narrative, register, portfolio, showcase, work, gallery, restraint

A portfolio exists so the work can be judged, so the page's own design must be strong yet
subordinate — confident typography, generous framing, near-zero decoration. Required furniture: a
selected body of work (fewer, better), each with enough context to be understood; a clear statement
of what this person or studio does; a reachable contact. Exceptional versions curate ruthlessly,
give each piece an honest one-line context (problem, role, outcome), and hold one consistent frame
so the eye compares the work rather than the containers. It fails when the interface competes with
the work, when the grid shows everything ever made, or when projects are decorative thumbnails with
no explanation of what was actually done.

## Register: agency-studio — capability proven by evidence

tags: layout, narrative, register, agency, studio, capability, proof, clients

An agency or studio page must convince a buyer that this team can be trusted with a budget, so it
balances a strong point of view against hard evidence. Required furniture: a stated position (what
this studio believes and does, not "we craft digital experiences"); named client work with results;
services described as outcomes; the team as real people; a direct way to start a conversation.
Exceptional versions lead with the strongest piece of evidence rather than a manifesto, use numbers
where numbers exist, and let a distinctive voice do the differentiating. It fails on generic
capability lists, stock-photo teams, and claims of craft that the page's own execution contradicts.

## Register: ecommerce-product — the object, without friction

tags: layout, narrative, register, ecommerce, product, retail, buy, detail

An ecommerce product page must let someone examine an object and buy it, so imagery quality and
answered objections matter more than narrative. Required furniture: multiple honest views of the
product including scale and detail; price and availability visible immediately; materials,
dimensions and specifics; shipping and returns answered near the ask; a persistent purchase action.
Exceptional versions photograph the product in use and at true scale, answer the exact question that
stops a purchase (fit, size, durability, care) beside the buy control, and let the object rather
than the layout carry the drama. It fails with one hero shot and no detail, with price hidden below
the fold, or with lifestyle imagery that never shows the thing being sold.

## Register: developer-tool — the page is documentation with a point of view

tags: layout, narrative, register, developer, technical, docs, code, precision

A developer tool page is read by someone deciding whether to spend an afternoon integrating, so
precision, real code, and honest scope beat marketing entirely. Required furniture: a one-sentence
statement of exactly what the tool does; real, copyable code showing genuine usage; installation in
one visible step; honest constraints and requirements; direct links to docs and source. Exceptional
versions show a real snippet in the first viewport, use monospace and dense information layout
without becoming unreadable, and state what the tool does NOT do. It fails with abstract 3D
illustrations instead of code, with benchmark claims lacking method, or with a marketing voice that
signals the writer never used it.

## Register: event-launch — a single moment with a deadline

tags: layout, narrative, register, event, launch, campaign, date, urgency

An event or launch page is organised around one date and one action, so hierarchy is unusually
steep: the what, when, where and the button outrank everything else. Required furniture: the name
and one-line premise; date, time and location unmissable; the single action (register, buy, join);
enough programme or lineup detail to justify committing; practical logistics. Exceptional versions
treat the date as a typographic anchor, use one bold visual idea consistently, and answer "is this
for me" within the first screen. It fails when atmosphere obscures the date, when the action appears
only at the bottom, or when the page reads as a brand piece that forgot to tell you when it happens.
